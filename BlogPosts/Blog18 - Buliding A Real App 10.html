My First Steps to Making My App Professional and User-Friendly

I've added two pieces of functionality that I think begin to take my project out of a personal project into something that users will need:

  1. Deleting folders and everything inside them safely in Firestore
  2. Creating a dynamic onboarding tooltip system
Both ended up teaching me a lot about atomic operations, observers, and thinking more carefully about UI state.

Deleting a Folder the Right Way: Learning to Use writeBatch

When I first implemented folder deletion, I took a straightforward approach: Loop through assignments → delete them one by one → update the student document → remove the folder.

It looked something like this: 


						for (const docSnapshot of snapshot.docs) {
							await deleteDoc(doc(db, "assignments", docSnapshot.id));

							await updateDoc(studentDocRef, {
								assignmentDocIds: arrayRemove(docSnapshot.id)
							});
						}

						await updateDoc(studentDocRef, {
							folders: arrayRemove(folderName)
						})
However, while testing out my app and its functionality, I noticed that I was able to delete a batch of items and interrupt this batch, leaving me with some deleted items and some items not yet deleted. I don't think this is really how it should be done on professional apps, so I researched what I could do and found writeBatch. 

						const batch = writeBatch(db)

						// Delete all assignments in the folder from assignments collection and remove their IDs from student doc
						snapshot.forEach((docSnapshot) => {
							batch.delete(docSnapshot.ref);
							batch.update(studentDocRef, {
								assignmentDocIds: arrayRemove(docSnapshot.id),
							});
						});

						// Remove folder name from student doc
						batch.update(studentDocRef, {
							folders: arrayRemove(folderName),
						});

						// Commit all at once
						await batch.commit();

What writeBatch Does

writeBatch creates a group of write operations that execute atomically. That means: Either all operations succeed, or none of them are applied. There is no in-between state.

With batching, Firestore guarantees consistency. This is important because folder deletion is a multi-document operation:

  • Delete multiple assignment documents
  • Update an array of assignment IDs
  • Remove the folder name from the student doc

Additional Information about writeBatch

One thing I learned is that writeBatch is write-only. You cannot:

  • Read Data
  • Check conditions
  • Perform logic based on current values
You must already know what you intend to update. In my case, that's fine because: I already queried the assignments beforehand, I know which IDs need to be removed, and I know the folder name being deleted But if I needed to check something dynamically during the write phase, writeBatch wouldn't be enough.

What is runTransaction?

While researching, I also learned about runTransaction. Unlike writeBatch, a transaction: Reads and writes, and amatically retries if data changes mid-operation

Roughly, runTransaction works like this:

  1. Read document(s)
  2. Compute updates
  3. Attempt write
  4. If data changed since step 1 → retry
  5. Repeat until success or failure
This makes transactions ideal when you need the latest document state or Multiple users may modify the same document. This could be something I use in the future as I possibly include multiple consultants to see a single student's account or allow for different levels of consultants, but for now I'll stick with writeBatch.

Building a Tooltip System (And Learning About the DOM)

The second big learning moment was building an onboarding tooltip system. Here's my Tooltip component: 


						const Tooltip = ({ targetElement, content }) => {

							if (!targetElement) return null;
							const rect = targetElement.getBoundingClientRect()

							return (
								<div
									className="tooltip"
									style={{
										position: "absolute",
										top: rect.bottom + 10,
										left: rect.left,
										background: "white",
										padding: "1rem",
										borderRadius: "8px",
										zIndex: 10001,
										boxShadow: "0 2px 8px rgba(0,0,0,0.3)"
									}}
								>
									<p>{content}</p>
									<div className="flex gap-2 mt-2">
										<button>Next</button>
										<button>Skip</button>
									</div>
								</div>
							)
						}
getBoundingClientRect() is a built-in DOM method available on any HTML element. It returns a DOMRect object. It returns the element's position and size relative to the viewport. It gives you: top, bottom, left, right, width, height.

So when I position the tooltip with top: rect.bottom + 10, I'm saying: Place this tooltip 10px below the target element.

With getBoundingClientRect(), I have a reliable way to dynamically position an overlay relative to an arbitrary DOM element. I don't have to worry about elements having been created already, or a user could be scrolling, or the position of parent elements.

Introducing the OnboardingManager

After I built my tooltip, I needed to figure out how to display my tooltip properly. It needs to show/know:

  • Which step we're on
  • Which page we're on
  • Whether onboarding is complete
  • Finding the correct target element
Dynamic Elements were an issue as I was building this out. Initially, I tried using querySelector, but if the element didn't exist yet (like when creating assignments dynamically), querySelector returned null. And React rendered nothing, permanently. I added a setInterval to continuously search for the correct selector but I felt this wasn't really the correct approach to just constantly search for an element.

The Better Solution: MutationObserver

After doing some research, I learned about MutationObserver, and changed my code to this: 


						const observer = new MutationObserver(() => {
							const element = document.querySelector(step.target) as HTMLElement | null;
							if (element) {
								setTargetElement(element);
								observer.disconnect();
							}
						});

						observer.observe(document.body, {
							childList: true,
							subtree: true
						});
MutationObserver is a browser API that: watches the DOM, reacts when changes occur, notifies you when elements are added/removed. So, instead of checking every 100ms, it just tells me when a change in the DOM has happened. As soon as the target element appears, the observer catches it and attaches the tooltip.

Now, my manager can just focus on finding the element and progressing through the steps of the onboarding process for my user. 

						const OnboardingManager = () => {
						const dispatch = useDispatch();
						const pathname = usePathname();
						const [targetElement, setTargetElement] = useState(null);
						const user = useSelector((state: RootState) => state.user);
						const { isComplete, onboardingStep } = useSelector((state: RootState) => state.onboarding);

						// Get step data
						const step = onboardingSteps[onboardingStep];
						
						useEffect(() => {
							if (!step?.target) {
								setTargetElement(null);
								return;
							}

							// First, try to find the element immediately
							const immediateElement = document.querySelector(step.target) as HTMLElement | null;
							if (immediateElement) {
								setTargetElement(immediateElement);
								return;
							}

							// If not found, set up MutationObserver to watch for it
							const observer = new MutationObserver(() => {
								const element = document.querySelector(step.target) as HTMLElement | null;
								if (element) {
									setTargetElement(element);
									observer.disconnect(); // Stop observing once found
								}
							});

							// Start observing changes to the document
							observer.observe(document.body, {
								childList: true,
								subtree: true
							});

							// Cleanup function
							return () => {
								observer.disconnect();
							};
						}, [step?.target, pathname]);

						// Validate step exists and has required properties
						if (!step || !step.target || !step.content || !step.page || !step.path) {

							return null;
						}

						const pathMatches = (stepPath: string, currentPath: string): boolean => {
							return currentPath.startsWith(stepPath);
						};

						if (!pathMatches(step.path, pathname)) {
							return null;
						} 


						if (onboardingStep >= onboardingSteps.length) {
							return null;
						}

						// If target element doesn't exist, don't render
						if (!targetElement) {
							return null;
						}

						const handleOnSkip = () => {
							dispatch(skip());
						}

						const handleOnNext = () => {
							dispatch(next())
						}

						return createPortal(
							<div className="onboarding-overlay">
								<Tooltip
									targetElement={targetElement}
									content={step.content}
									onNext={handleOnNext}
									onSkip={handleOnSkip}
									page={step.page}
								/>
							</div>,
							document.body
						)
					}

					export default OnboardingManager

Final Takeaways

This round of coding was a bit frustrating. I feel like I didn't get a TON of functionality done since it took a while to debug the rendering loop. However, I did get some cleanup done with my code by centralizing my dispatches, so it'll be easier to debug in the future. And, I did get a core feature put in. So, a win's a win.

Lessons Learned

  • Batch writes protect against partial failures
  • runTransaction is for conditional read-modify-write logic that I may have to go to later
  • Rendering depends on DOM timing more than expected
  • getBoundingClientRect() unlocks positioning flexibility
