Building Secure Student Account Creation with Firebase Admin and Fixing a Big Auth Problem

I ran into a major problem when implementing Firestore security rules and account creation for students. Initially, the rules for creating a student seemed straightforward:

  1. Consultants can create student accounts and then manage those accounts
  2. Students can only read and update their own data
My Firestore Database Rules has has two main user collections:
  1. consultantUsers
  2. studentUsers
My rules feel pretty straightforward. For consultants who need to access their own data freely: 

						match /consultantUsers/{userId} {
							allow read, update, delete: if request.auth != null && request.auth.uid == userId;
							allow create: if request.auth != null;
						}
For students, I have two access rules:
  1. Students can read/update their own data
  2. Consultants can read/update their assigned students 

						match /studentUsers/{userId} {
						allow read, update: if request.auth != null &&
							(request.auth.uid == userId || request.auth.uid == resource.data.consultant);

						allow delete, create: if request.auth != null;
						}
After adding in the functionality and rules to create students, I ran into my problem. After creating a student, my view of my dashboard had changed. It was returning an unauthorized page. After implementing my rules... I wasn't authorized to see my dashboard?... That wasn't right. I kept refreshing and that wouldn't work either.

I logged out and back in, and suddenly I could see my dashboard page. It took quite a bit of playing around with what was going on, but I eventually figured out that creating a student was causing me to see the unauthorized page after, then after a lot of debugging, I realized that my userId was no longer the same after creating a student. That's when I found out that createUserWithEmailAndPassword logs you into your account automatically.

The core of the issue


						createUserWithEmailAndPassword(auth, email, password)
I loved this firebase function since it was short and sweet and handled all the user creation with just a simple line. But now it was signing me out as the consultant and signing me in as a student, which obviously is a problem. I guess this was a problem the entire time, but I hadn't seen it until my rules had been implemented to restrict who sees what.

My Solution: Server-Side User Creation

After some research, I found that I can create users on my own in the backend using the Firebase Admin SDK. Using this, I could create my users without switching auth, but it also came with the fact that I would bypass client-side security restrictions so I had to make sure to use auth tokens. I hadn't don't auth in some time, so I had to research a bit more about what to do there as a refresher.

Creating an API Route for Student Creation

First, I created a Firebase Admin instance with some boilerplate code, then I added a Next.js API route.

Overall, calling this API route does the following:

  1. Receive the student data
  2. Verify the consultant's auth token
  3. Create the student account
  4. Create the student document
  5. Link the student to the consultant
Since I had to dive into auth and how it works again, I'll talk a bit about the process to verify the consultant:

The frontend sends the consultant's ID token in the request header. 

						const authHeader = request.headers.get('authorization');

						if (!authHeader?.startsWith('Bearer ')) {
						return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
						}

						const token = authHeader.split('Bearer ')[1];
						const decodedToken = await adminAuth.verifyIdToken(token);
This guarantees the request is coming from an authenticated Firebase user. Then I confirm the user is actually the consultant making the request: 

						if (decodedToken.uid !== consultantId) {
						return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
						}

Creating the Student Account

Now the server safely creates the user: 


						const userRecord = await adminAuth.createUser({
						email,
						password,
						emailVerified: false,
						});


So now I'm capable of creating users without having to log out from my current user and changing my auth session.

To talk a bit more about what's happening in Firestore, I create a student with a few different fields. Some of this stuff probably won't stick forever since the app will evolve, but for now here is what I have:

After creating the account, I store the student data. 

						const studentRef = adminDb.collection('studentUsers').doc(userRecord.uid);

						await studentRef.set({
						personalInformation,
						academicInformation,
						consultant: consultantId,
						folders: folders || [],
						email,
						createdAt: new Date().toISOString(),
						});
Then I update the consultant document to include the student. 

						await consultantRef.update({
							students: admin.firestore.FieldValue.arrayUnion(studentRef)
						});
In the future, I'll probably add more to these fields when a student is created, like adding metadata for users on their assignments or allowing for multiple consultants to connect with a single student for any collaboration.

Since I was working on sigin-up and implementing database rules, I also added email verification.

Improving Performance

After deploying this feature to real users, I did have some notes about how slow it was to log in and create student accounts, about 5ish seconds. So I started to research how to optimize my app.

Running Firestore Operations in Parallel. Originally, my code ran sequential operations:

  1. Create student document
  2. Update consultant document
While I was creating functionality, I wasn't quite refactoring along the way with speed in mind. I just wanted to get the functionality and pass my tests. Now that this issue has come up, I decided to use Promise.all() to run my calls in parallel. 

						await Promise.all([
						studentRef.set(studentData),
							consultantRef.update({
								students: admin.firestore.FieldValue.arrayUnion(studentRef)
							})
						])
With my own tests, student creation went from a 5 second wait to about 3 seconds. Obviously I'm not at some enterprise level, but I don't think that's the worst amount of time to wait, especially with functionality that won't be used daily by users.

Improving Login Speed

I also had another inefficiency during login. To determine whether a user was a consultant or student, I originally did two database lookups: 


					const consultantDoc = await getDoc(doc(db, "consultantUsers", userId))

					const studentDoc = await getDoc(doc(db, "studentUsers", userId))
Again, it works, but now that I have users testing and speed is becoming an issue, as well as this being something users will do every time they want to use the app, I want to get rid of any extra latency.

I switched to using Firebase custom claims, allowing me to add metadata to the auth tokens. Now, I can determine a role from my user based on their token that will direct the lookup in the proper table right away, rather than looking up both.

This also prompted me to make some other changes with login that would make users feel loading was happening faster. Previously, the order I had called was:
  1. Login
  2. Fetch user data
  3. Redirect
This was causing us to stay on the login page for an extended time period.

Now:
  1. Login
  2. Redirect immediately
  3. Fetch data after page load
Now the fetch for the user will happen on the dashboard page rather than the login page. Even though the actual load time is similar, the user perceives the app as much faster.

Final Takeaways

This round of coding was a bit frustrating. I feel like I didn't get a TON of functionality done since it took a while to debug the rendering loop. However, I did get some cleanup done with my code by centralizing my dispatches, so it'll be easier to debug in the future. And, I did get a core feature put in. So, a win's a win.

Lessons Learned

This experience taught me several important backend lessons.

  1. Develop platforms can have underlying functionality with their easy to use functions, so make sure you have a clear understanding of whats going on.
  2. Parallelizing operations matters - Running operations concurrently can noticeably improve performance. Running batches in Firestore is also another way to make sure that all our data is logged, or none of it is, stopping us from having incomplete information added to our database.
  3. Reduce database lookups whenever possible - Now that I have users on the app, I don't want to slow down their experience or push me to use a paid subscription too early. Use any tools available, like custom auth claims, to eliminate unnecessary queries.
  4. Perceived speed matters - Something that shows progress to the user is always good, even if it is purely frontend without the fetched data.

What's Next
