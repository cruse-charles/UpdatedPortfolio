See how I build a web app as I learn a new technology

February 13, 2025

What I Did This Week

This week's progress on my Learn-to-Code platform included a mix of small but important refactors, styling, and researching for a major feature upgrade: recording a user's screen and webcam as they solve problems.

Fixing Example Rendering Logic

Originally, my DSA problems in MongoDB stored example information as arrays inside a descriptionExamples object. So in my frontend, I looped through the input, output, and explanation arrays by index: 


						// Examples section, iterate through example information
						
Examples:

						{descriptionExamples.input.map((_, index) => (
						<div className='mb-4 bg-slate-50 p-4 rounded-md' key={index}>
							<div className="mb-2">
							<div>Input: {descriptionExamples.input[index]}</div>
							</div>
							<div className="mb-2">
							<div>Output: {descriptionExamples.output[index]}</div>
							</div>
							<div className="mb-2">
							<div>Explanation: {descriptionExamples.explanation[index]}</div>
							</div>
						</div>
						))}
It worked, but as I added more problems, I realized this structure was both awkward to iterate through and prone to bugs if the array lengths got out of sync. It also made updates in MongoDB a pain.

I refactored the data structure to use an array of objects instead:


					"descriptionExamples": [
					{
						"input": "s = '()'",
						"output": "true",
						"explanation": "The parentheses are matched correctly."
					},
					...
					]

Then in the frontend, it was much cleaner:


					{descriptionExamples.map((example) => (
					<div className='mb-4 bg-slate-50 p-4 rounded-md'>
						<div>Input: {example.input}</div>
						<div>Output: {example.output}</div>
						<div>Explanation: {example.explanation}</div>
					</div>
					))}

Changing the database format was a bit tedious, but thankfully I only had four questions to update. Definitely the right call in the long run.

Checking Output and Providing Feedback

Another win this week was finishing up the code submission UI. Now when users write a solution and click “Run Code,” the system:

  1. Runs the function with multiple test cases.
  2. Checks the results against expected outputs.
  3. Shows clear success/failure feedback.
Right now, Im showing a simple pass/fail per case, but I plan to enhance it with:
  • User console.log() output
  • Runtime errors with error messages
  • Possibly syntax highlighting for failed outputs
The feedback loop is almost complete, and it's already feeling satisfying to use.

Planning a Recording Feature for Live Practice

Here's the next big goal: Add a "Record" button to let users film themselves solving problems — both screen and webcam.

The vision is simple but hopefully powerful in helping people see how they act and explain their code instead of just what their code is. It's part mock interview practice, part personal coaching tool. High level, it'lls do the following:

  • You press record.
  • Your screen and webcam are captured as you solve a coding problem.
  • The video is saved locally.
  • You can watch your performance, observe your communication style, debug habits, and even review how you talk through problems.
I'm mostly doing this because in real interviews, it's not just about code, it;s how you think, speak, and carry yourself as well. Recording that process gives users a way to analyze and improve their overall performance, not just their algorithm skills.

Choosing a Recording Approach: MediaRecorder API

I've started looking into how to implement this, and the MediaRecorder API looks like a solid path for a few reasons:

  • Native to the browser, no extra packages required
  • Supports both webcam (navigator.mediaDevices.getUserMedia) and screen (navigator.mediaDevices.getDisplayMedia)
  • Allows saving to .webm or .mp4 and downloading to the user's computer
