When Security Rules Break Your App

To prepare for actual users and ensure my app is not vulnerable to anyone malicious, I started tightening my Firestore security rules. Unfortunately, doing so broke a core functionality of my app

The Rule That Made Sense But Later Didn't

My system has two user types: Consultants and Students. Consultants should be able to:

  1. Create student accounts
  2. Manage their students
  3. Read and update student data


So I wrote my Firebase database security rules like this: Consultants should be able to read and update their own students: 

						match /studentUsers/{userId} {

						allow read, update: if request.auth != null &&
							(request.auth.uid == userId || request.auth.uid == resource.data.consultant);

						allow delete, create: if request.auth != null;
						}
For assignments: 

						match /assignments/{assignmentId} {

						allow read: if request.auth != null && (
							resource.data.consultantId == request.auth.uid ||
							resource.data.studentId == request.auth.uid
						);

						allow create: if request.auth != null &&
							request.resource.data.consultantId == request.auth.uid;

						allow update, delete: if request.auth != null &&
							resource.data.consultantId == request.auth.uid;
						}
This felt straightforward and easy until I implemented the rules and started testing CRUD functionality on creating a student as a consultant.

The Core Problem: createUserWithEmailAndPassword

Originally, I was using Firebase functions to create my users. It was quick and easy to use as I just needed to pass a user's credentials into a function and then I have a new user. 


						const userCredentials = await createUserWithEmailAndPassword(
							auth,
							formData.email,
							formData.password
						);
However, after creating a user, I was sent to my unauthorized page and unable to view my dashboard or anything else. Logging in and out allowed me to work as normal, and logging in as the student was fine too, so why did it happen when I created a student? It took some time to research this issue, but I found out that Firebase automatically logs in as the newly created user.

That means:
  • Consultant clicks “Create Student”
  • Firebase creates the student
  • The app is now authenticated as the student
Which means:
  • request.auth.uid is now the students ID
  • My frontend auth state is now wrong
  • I need to re-login after every student I create
I obviously had to keep the security logic, so this meant I had to change how I was designing my app.

The Solution: Use the Firebase Admin SDK

From what I was reading, the solution was to use Firebase Admin SDK, which is a server-side library that lets your backend act as a trusted environment.

By using Firebase Admin, I can:

  • Bypasses Firestore security rules
  • Keep my authentication context
  • Implement my own rules for requests
So I installed the firebase-admin SDK and created a server instance using their boiler-plate code. 

						import admin from "firebase-admin";

						if (!admin.apps.length) {
						admin.initializeApp({
							credential: admin.credential.cert({
							projectId: process.env.FIREBASE_PROJECT_ID,
							clientEmail: process.env.FIREBASE_CLIENT_EMAIL,
							privateKey: process.env.FIREBASE_PRIVATE_KEY?.replace(/\\n/g, "\n"),
							}),
						});
						}

						const adminAuth = admin.auth();
						const adminDb = admin.firestore();

						export { admin, adminDb, adminAuth };

Creating a Secure API Route

Now that I could use Firebase Admin to create my own sort of rules and requests, I created a Next.js API route (some code is cut out to be concise): 


						export async function POST(request: NextRequest) {
							try {
								// Parse the request body for details for the new student account
								const { email, password, personalInformation, academicInformation, folders, consultantId } = await request.json();

								// Create a new Firebase Authentication user for the student
								const userRecord = await adminAuth.createUser({
								email,
								password,
								emailVerified: false, // Student email not verified yet
								});

								// Reference to the Firestore document for the new student
								const studentRef = adminDb.collection('studentUsers').doc(userRecord.uid);

								// Create the student document in Firestore with the provided details and link the student to their consultant
								await studentRef.set({
								personalInformation,
								academicInformation,
								consultant: consultantId,
								folders: folders || [],
								email,
								createdAt: new Date().toISOString(),
								});

								// Reference to the consultant's Firestore document
								const consultantRef = adminDb.collection('consultantUsers').doc(consultantId);

								// Add the new student's document reference to the consultant's "students" array
								await consultantRef.update({
								students: admin.firestore.FieldValue.arrayUnion(studentRef)
								});

								// Return success response with the new student's UID
								return NextResponse.json({
								success: true,
								studentId: userRecord.uid,
								message: 'Student created successfully'
								});

							} catch (error: any) {
								// Return 500 Internal Server Error with the error message
								return NextResponse.json({ error: error.message }, { status: 500 });
							}
						}
Now I'm able to create students and still keep the correct auth context.

Lessons Learned

I also added email verification during login. Two things I learned:

  1. On the free tier, you can't fully customize the email template.
  2. The verification URL Firebase sends includes query parameters that:
    • Contain the mode (verifyEmail)
    • Include an action code
    • Support redirect URLs
So now I can choose to either:
  • Send users back to your app
  • Display success/failure messages
  • Trigger backend updates
Since I started coding, I've been looking more into URLs that I encounter to see what they display, but hadn't thought much about the links sent in verification emails.

Final Takeaways

  • Firestore functions and rules are great but are limited
  • The Admin SDK bypasses rules intentionally, so it has to be used carefully
  • I need to brush up more on security standards and rules now that I have to implement some of my own and want to have real users
