Building A Real App #9

Follow along with my journey to build a real app

July 10, 2025

Building a Scalable Assignments View

As my app grew beyond a simple list of assignments, I started running into problems in React re-renders, Redux state management, and component responsibility.

This post walks through what I implemented, what broke, why it broke, and what I learned fixing it.

Adding Sorting to Assignments and Folders

I wanted to allow users to sort assignments inside folders, and also sort folders themselves. Two sets of sorting, one within another. It took a bit of time to figure out the logic to it.

Sorting Assignments Within a Folder

Assignments already belonged to folders, so the first step was filtering assignments by folder and then applying a sort based on user selection.

At first, this felt tricky because the assignments list was already filtered, and I was wondering how to implement rendering the different sort types, but any function that runs during render will re-run automatically when state changes. So I didn't have to worry about managing how it renders. As long as the sorting that I was choosing (assignmentSort) was state-driven, the sorting logic would naturally re-run.

Here's the function I ended up with: 


						const getFilteredAssignments = (folder: string) => {
							if (!assignments) return []

							let folderAssignments = assignments.filter(
								(assignment) => assignment.folder === folder
							)

							if (assignmentSort === 'name') {
								folderAssignments.sort((a, b) => a.title.localeCompare(b.title))
							}

							if (assignmentSort === 'due') {
								folderAssignments.sort((a, b) => {
									const dateA =
										a.dueDate instanceof Timestamp
											? a.dueDate.toDate()
											: (a.dueDate as Date)
									const dateB =
										b.dueDate instanceof Timestamp
											? b.dueDate.toDate()
											: (b.dueDate as Date)

									return dateA.getTime() - dateB.getTime()
								})
							}

							if (assignmentSort === 'status') {
								const statusOrder = [
									'Overdue',
									'Pending',
									'Submitted',
									'Under Review',
									'Completed',
								]
								folderAssignments.sort(
									(a, b) =>
										statusOrder.indexOf(a.status) -
										statusOrder.indexOf(b.status)
								)
							}

							return folderAssignments
						}
This worked well and stayed readable. Sorting inside a filtered dataset turned out to be much simpler than I initially expected.

Sorting Folders by Closest Due Date

Sorting folders by their own name was easy, but it was more challenging to sort them by an object within the folder instead.

So the solution required a “ride-along” value: calculating something inside the sort comparison that never actually gets returned. This was a new concept to me but it's stuck with me and I'm looking forward to using it in other kinds of sorting that I try to implement too.

The idea was:

  1. Get all assignments for a folder
  2. Extract their due dates
  3. Ignore past dates
  4. Find the soonest upcoming one
  5. Sort folders by that value
Here's the final version: 

						const sortFolders = () => {
							let sortedFolders = [...folders]

							if (folderSort === 'name') {
								sortedFolders.sort((a, b) => a.localeCompare(b))
							}

							if (folderSort === 'due') {
								const now = new Date()

								sortedFolders.sort((a, b) => {
									const assignmentsA = getFilteredAssignments(a)
									const assignmentsB = getFilteredAssignments(b)

									const soonestDueDateA = assignmentsA
										.map((assignment) =>
											assignment.dueDate instanceof Timestamp
												? assignment.dueDate.toDate()
												: (assignment.dueDate as Date)
										)
										.filter((date) => date >= now)
										.sort((d1, d2) => d1.getTime() - d2.getTime())[0]

									const soonestDueDateB = assignmentsB
										.map((assignment) =>
											assignment.dueDate instanceof Timestamp
												? assignment.dueDate.toDate()
												: (assignment.dueDate as Date)
										)
										.filter((date) => date >= now)
										.sort((d1, d2) => d1.getTime() - d2.getTime())[0]

									if (!soonestDueDateA && !soonestDueDateB) return 0
									if (!soonestDueDateA) return 1
									if (!soonestDueDateB) return -1

									return (
										soonestDueDateA.getTime() -
										soonestDueDateB.getTime()
									)
								})
							}

							return sortedFolders
						}
“Ride Along” behavior starts where we are given a folderSort that is 'due.'' The folders are still the primary objects being sorted, but their relative order is decided using data from their assignments.

For each comparison, the sort callback:
  • Takes two folders (a and b)
  • Fetches the assignments associated with each folder


Each folder's assignments are then:
  1. Converted into Date objects
  2. Filtered to exclude past due dates
  3. Sorted chronologically, and the earliest one is selected.
This gives each folder a single representative value: its next due assignment, which will order folders by their soonest due date.

So, in this function, assignments “ride along” with the folders. The assignments are not sorted or changed in any way, just the folders are reordered based on this value we extract to represent the folder.

The Infinite Render Loop That Broke Everything

At one point, the app became nearly unusable:

  • Student profiles wouldn't load
  • Pages crashed unless refreshed
  • React threw “maximum update depth exceeded”
Refreshing fixed it, which immediately told me this was a state lifecycle problem.

What Was Actually Happening

The issue wasn't Redux alone. It was Redux + local state syncing. With debugging, I found the issue was working like this:

  • The page dispatches fetchStudent(studentId)
  • Redux updates the student's folders
  • A component runs useEffect(() => setFolders(reduxFolders), [reduxFolders])
  • setFolders triggers a re-render
  • Redux selectors return a new array reference
  • The effect runs again infinitely
The core issue was copying Redux state into local state unconditionally. Once I stopped doing that, and instead derived everything directly from Redux, the problem disappeared.

Centralizing Data Fetching at the Page Level

Another mistake became obvious while debugging: I was dispatching Redux actions from multiple components that all lived on the same page. That made the data flow hard to trace since I was putting in console.logs that weren't appearing.

I refactored so that:

  • All data fetching happens in the page component, so I don't have to manage 10 files for one functionality
  • Child components only read from Redux
  • No child initializes or mutates shared state


Here's the cleaned-up structure all in one easy to find spot: 

						useEffect(() => {
						dispatch(clearAssignments())
						dispatch(fetchStudent(studentId))
					}, [studentId, dispatch])

					useEffect(() => {
						if (student?.assignmentDocIds) {
							dispatch(fetchAssignments(student.assignmentDocIds))
						}
					}, [student?.assignmentDocIds, dispatch])

Final Takeaways

This round of coding was a bit frustrating. I feel like I didn't get a TON of functionality done since it took a while to debug the rendering loop. However, I did get some cleanup done with my code by centralizing my dispatches, so it'll be easier to debug in the future. And, I did get a core feature put in. So, a win's a win.

Lessons Learned

  • How to do more complex sorting based on values 'within' an object instead of the object itself
  • Syncing Redux state into local state can easily cause render loops
  • Centralizing Redux dispatches at the page level improves clarity (and debugging)
