Building A Real App #14

Follow along with my journey to build a real app

Oct 15, 2025

Refactoring AssignmentList + Fixing Redux Persistence Bugs

As my application grew into a real tool used by education consultants, the AssignmentList component became a bottleneck.

It originally started as a simple UI layer, but gradually became a file that held functionality from various parts of the entire system:

  1. assignment creation
  2. nested folder sorting
  3. notifications
  4. onboarding hooks
  5. modal handling
  6. status updates
  7. read/unread tracking

It eventually hit ~400 lines. So I finally refactored it.

Breaking Down a 400-Line Component Into Modules

The first major change was structural. I split the monolith into smaller pieces by functionality:

  • AssignmentRow
  • FolderRow
  • SortControls
  • useAssignmentSort hook
This reduced AssignmentList to about 100 lines. Now:

AssignmentRow handles:
  • Rendering an assignment
  • Click behavior
  • Local UI state
FolderRow handles:
  • Folder expansion
  • Nested assignment rendering
SortControls handles:
  • Sorting UI
  • Filter state
useAssignmentSort handles:
  • Sorting logic (pure function-like behavior)

The Unexpected Problem: “Where Does Logic Belong?”

During refactoring, I hit a confusing architectural issue. Some logic needed to stay in the parent component, like modal state coordination, global assignment selection, onboarding triggers, and cross-component updates.

However, this also made me re-think if I should keep logic closer to its UI or keep it all central in the parent component for easier coordination.

I decided to accept a hybrid model: Local components for UI and Parent component for orchestration

Redux Refactor

One major pain point was passing assignmentClick through multiple component layers, from AssignmentList, to Folders to Assignments, to eventually open an assignmentModal. So I moved modal state into Redux: 


						const currentAssignmentSlice = createSlice({
							name: 'currentAssignment',
							initialState,
							reducers: {
								setCurrentAssignment(state, action) {
									state.assignment = action.payload
								},
								openCurrentAssignmentModal(state) {
									state.isModalOpen = true
								},
								closeCurrentAssignmentModal(state) {
									state.isModalOpen = false
								}
							}
						})
Now I can open modals from anywhere with one function and remove a lot of props. However, this led to a new problem…

Production Login Suddenly Failed

After deploying this new feature, I was stuck with a very weird bug when I asked test users to re-try my app. My personal local dev worked fine, but the live site failed after login. Why could I log into my own account fine on a local dev environment but not on the live website?

Even more confusing, new users worked fine on the live site, it was only existing users that were broken. It took me a while to figure out the issue.

The issue was Redux persist. I changed the shape of my Redux state:

  • CurrentAssignment became an object wrapper
  • Nested structure changed
  • Fields were renamed


I double checked by signing into the live site with my old account in incognito mode, where I wouldn't be pulling from old redux, and logging in worked fine. So, since I persisted the old Redux state in local storage, an error would appear upon login where the old redux structure would load and wouldn't match the new structure that was being brought in.

The Fix: Purging Persisted State

Instead of removing persistence entirely (which I still needed for UX), I added persistor.purge().

This would trigger on the login page and while logging out. Now, redux resets cleanly and there is no state collision.

Firestore Rules Became More Complex

Because of structural changes, I also had to update security rules. Instead of repeating logic everywhere, I introduced: 


						function isConsultantOfStudent(userId) {
						return get(/databases/$(database)/documents/studentUsers/$(userId))
							.data.system.consultantId == request.auth.uid;
						}
Then used it across rules: 

						allow read, write: if request.auth != null && (
							request.auth.uid == userId ||
							isConsultantOfStudent(userId)
						);

Lessons Learned

  1. Refactoring uncovers so many issues when separating files and functionality, do it often so you don't have to do a huge overhaul.
  2. Take a long-term approach to schema design. It can always be 'incomplete' and just add a few fields, but if you start really changing a lot of the structure, you'll have to change a lot in the codebase. Think of everything you might need in the beginning, and stick with that.

What's Next

Refactoring everything was a bit of fun honestly. Making things more efficient and figuring out how to change a lot of different parts of the codebase to adjust for this does make me feel like I'm creating a real app, so that's a ton of fun. I'll definitely continue refactoring more often to make sure there isn't a huge build up like this time though!
