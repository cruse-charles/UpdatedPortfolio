Building A Real App #12

Follow along with my journey to build a real app

Sep 12, 2025

Building on my Onboarding System Again and Enhancing My Website

As my app started becoming more usable with the core features, I started running into a problem with my users. Obviously, I know how the app should work, and I feel it isn't too complex, but I had a lot of test users prove otherwise.

Before building my first version of the onboarding system, testers would just sit and stare at the screen without doing anything, even though I told them that I would not instruct them to use the app. So, working on this feeling of users unwilling to explore themselves, I built an onboarding system.

A pop-up would appear next to the buttons that I wanted them to click. Creating students, assignments, and checking out other parts of the website. However, even that proved not to be enough. After having users test the website again with my first version on the onboarding website, I saw users just rapidly clicking through “Next” button of the onboarding without even reading anything.

So, I had an app that I knew solved a problem for a lot of these kinds of users, but I still wasn't able to get them to use it properly… that made me rethink how to implement the onboarding system. Instead of just notifying them of the functionality, I decided to force them to use the functionality throughout onboarding.

This was a big lesson in learning to build things to be adaptable to future code editing, reviewing my CSS and DOM knowledge, and getting into the mindset of a range of users.

This post walks through:

  1. Fixing fragile onboarding logic
  2. Making progression action-based instead of index-based
  3. Using Promise.all for performance improvements
  4. Building a dynamic onboarding system with DOM targeting + portals
  5. Creating an animated highlight system with CSS
  6. Designing for real user behavior (not ideal users)

The First Problem: Fragile Onboarding Logic

Originally, my onboarding progression logic looked like this. If the user isn't complete with their onboarding, then we want to dispatch the needed step on the respective action: 


						if (!isComplete) {
							dispatch(completeStep('createEntry'))
							await nextStep(user.id)
						}
At first glance, this works. But that assumes the user is going in order. A big flaw was that they could click on another 'action button' by accident. This would push their onboarding forward even though they didn't do what was needed.

I also had a big issue with hardcoding which numbered step was being called for my first fix. If I wanted to add an additional step, then I had to renumber everything in my list of actions and renumber everything on my frontend to reference the correct number.

The Fix: Action-Based Progression

Instead of tying logic to step numbers, I refactored everything to depend on a unique action identifier. Each step defines what action is required: 


						{
							id: "add-student-btn",
							actionRequired: "clickAddStudentButton"
						}
Then in handlers: 

						const currentStep = onboardingSteps[onboardingStep].actionRequired

						if (!isComplete && currentStep === "fetchStudentProfile") {
							dispatch(completeStep("fetchStudentProfile"))
						}
Now:
  • Order doesn't matter as much
  • Adding new steps doesn't break existing logic
Updating across several different files and functions is eventually going to slip me up and I'll need to go back to fix it later. So, this was a good reminder to always code with scalability and flexibility in mind.

Example: Fixing a Click Handler

Here's one of the refactored handlers: 


						const handleStudentClick = async (studentId: string) => {
							if (!isComplete && onboardingStep === 2) {
								const currentStep = onboardingSteps[onboardingStep].actionRequired

								if (currentStep === "fetchStudentProfile") {
									dispatch(completeStep("fetchStudentProfile"))
								}

								await nextStep(user.id)
							} 
								
							router.push(`/consultant/students/${studentId}`);
						}
This ensures:
  • The step only progresses if the correct action happens
  • Random clicks don't break onboarding

Highlighting Elements (The Animated Border)

To really make sure users would take a look at what I wanted them to look at, I added a highlight box around the target element. I was able to get a dashed border around the target element, but I wanted it to pop a bit more by having the dashes kind of mover around the border. I asked ChatGPT for some help in finishing up the animation and came up with what is below: 


						const highlightStyles = {
							position: "absolute",
							top: rect.top + window.scrollY - 6,
							left: rect.left + window.scrollX - 6,
							width: rect.width + 12,
							height: rect.height + 12,
							borderRadius: "8px",
							pointerEvents: "none",
							zIndex: 10000,
						}
Animation: 

						@keyframes dash-move {
							to {
								background-position:
								32px 0,
								-32px 100%,
								0 -32px,
								100% 32px;
							}
						}

Performance Improvements: Promise.all

While fixing onboarding, I also optimized performance in other parts of the app, taking some inspiration from what I did last week after making login appear faster to the user.

I fetch user data when they log into their dashboard, for a quick peek at what needs to be done for the week and how things are progressing. Previously, I had each call go separately. Instead of doing async operations sequentially: 


						await countOfAssignmentsNotFinished()
						await nextAssignmentDue()
I switched to: 

						await Promise.all([
							countOfAssignmentsNotFinished(),
							nextAssignmentDue()
						])
Now: Both operations run at the same time, and total time waited is just the max of the operations rather than the cumulative amount.

This was also a help in making my dashboard page load faster. Previously, I was workin on UI improvements, like just immediately going to the dashboard page with loading states when logging in instead of waiting to login first. Now, I'm improving my calls for data too, and it definitely feels faster!

Lessons Learned

The biggest takeaway from this entire process: Users will not follow your intended flow. So, if you have a specific process you need them to complete, make sure you make it flexible. So your system must:

  1. handle out-of-order actions
  2. prevent invalid progression
  3. guide clearly and visually
